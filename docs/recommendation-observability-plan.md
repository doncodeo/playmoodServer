# Recommendation, Super Admin User Telemetry, and Continue Watching Review

## 1) Recommendation improvements

### Current behavior observed
- Homepage feed and related recommendations are generated by `recommendationService.getRecommendations`.
- Ranking currently mixes behavior signals (likes, watch progress, rewatches, hovers, comments, unfollows), content similarity, and popularity.
- We now support **admin-managed forced recommendations** through `ForcedRecommendation` records merged into recommendation output.

### Admin-managed recommendation mechanism (implemented foundation)
- New model: `ForcedRecommendation` with:
  - `contentId`
  - `priority`
  - `startsAt` / `endsAt`
  - `isActive`
  - `reason`
  - `createdBy`
- Service-level merge strategy:
  1. Pull active forced entries sorted by priority.
  2. Resolve to approved content.
  3. Inject at top of feed and dedupe with personalized results.

### Recommended next steps
1. Add admin CRUD endpoints for forced recs (create/list/update/deactivate).
2. Add audit trail and moderation notes.
3. Add targeting fields (country, platform, appVersion, cohort) for controlled overrides.
4. Add experiments/flags for ratio control (e.g., max 20-30% forced slots).

## 2) Super admin user telemetry (recommended architecture)

To track device/OS/location safely and compliantly:
1. Introduce a `UserSessionEvent` collection (immutable append-only events).
2. Capture from request metadata:
   - IP (store hashed/anonymized form where possible)
   - User-Agent raw + parsed (`device`, `os`, `browser`)
   - Optional client-provided app metadata (app version, build number)
   - Geo lookup (city/country/region) from trusted GeoIP service
3. Add retention policy (e.g., 30/90/180 days) and privacy controls.
4. Build super-admin aggregate endpoints (daily active users, devices, regions, OS mix), avoiding raw PII by default.

## 3) Continue Watching + share timestamp

### Current status observed
- Continue watching is sorted by `lastWatchedAt`, filtered to unfinished videos, capped at 20.
- Resume URLs include timestamp in the format `?t=<seconds>s`, e.g. `/movie/<slug>-<id>?t=50s`.

### Gap to be aware of
- This timestamped URL is currently emitted by backend for continue watching payload.
- If FE sharing flow builds URLs independently, ensure FE always appends `t` from current playback position and that player parser accepts both `t=50` and `t=50s` for resilience.

### Recommended optimization work
1. Add index strategy for `videoProgress.contentId` + `lastWatchedAt` patterns if growth becomes large.
2. Add a dedicated share-link endpoint as canonical URL generator (prevents FE drift).
3. Add E2E tests validating URL parse/seek for shared links.
